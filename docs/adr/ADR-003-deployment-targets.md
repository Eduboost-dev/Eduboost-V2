# ADR-003 — Deployment Targets and Promotion Model

**Status:** Accepted
**Date:** 2026-05-27 (Phase 1 T145)
**Decision owner:** Platform / Engineering
**Supersedes:** none

---

## Context

EduBoost V2 currently has multiple deployment targets and promotion paths.
This ADR documents each target, its purpose, and the canonical promotion
model from local development to production.

---

## Decision

### Target inventory

| Target | Runtime | Purpose | Ownership | SLA |
|---|---|---|---|---|
| **Local (Docker Compose)** | Docker Compose | Developer local development | Developer workstation | Best effort |
| **Render (Blueprint)** | Render web service | Staging / preview deployments | Render platform | Render SLA |
| **Kubernetes (production)** | k8s cluster | Production deployment | Platform team | 99.9% uptime |

### Promotion model

```
Local dev ──► PR branch ──► CI pass ──► Merge to main ──► Render staging
                                        │
                                        └─── Tag release ──► GitHub Release
                                                              │
                                                              └──► k8s production
```

1. **Local development:** `docker compose up --build`
2. **PR validation:** All CI gates must pass.
3. **Merge to `main`:** Auto-deploys to Render staging (if `autoDeployTrigger: commit` is enabled).
4. **Release tag:** Manual workflow dispatch of `release.yml`.
5. **Production:** `production-promote` job deploys to Kubernetes via `kubectl set image`.

---

## Target details

### 1. Local development (Docker Compose)

**File:** `docker-compose.yml`

Services:
- `postgres:15` — local database
- `redis:7` — local cache / sessions
- `api` — FastAPI app via `Dockerfile.v2`

**Entry:** `uvicorn app.api_v2:app --host 0.0.0.0 --port 8000`

**Use case:** Day-to-day development, debugging, integration testing.

**Limitations:**
- No SSL/TLS termination.
- No CDN or edge caching.
- No managed Redis persistence.
- No horizontal scaling.

### 2. Render (Blueprint)

**File:** `render.yaml`

Configuration:
- `runtime: python`
- `plan: free` (should be upgraded for production)
- `region: oregon`
- `branch: main`
- `healthCheckPath: /ready`
- `autoDeployTrigger: commit`

**Environment variables:**
- `DATABASE_URL` — Supabase Postgres (sync: false)
- `SECRET_KEY` — auto-generated by Render
- `REDIS_URL` / `UPSTASH_REDIS_REST_URL` — Upstash Redis

**Use case:**
- Staging environment for PR previews.
- Lightweight production for early beta.
- Fallback if k8s cluster is unavailable.

**Limitations:**
- Free plan has cold-start latency.
- No persistent disk (ephemeral filesystem).
- No custom domain on free plan.
- Single-region (Oregon).

### 3. Kubernetes (production)

**Deployment:** `docker/api-deployment.yml`

**CI promotion:** `ci-cd.yml` `production-promote` job

Steps:
1. Decode `KUBECONFIG_DATA` secret to `/tmp/kubeconfig`.
2. `kubectl set image deployment/eduboost-api api=<image:tag>`.
3. `kubectl rollout status deployment/eduboost-api --timeout=5m`.
4. Health check: `curl $PROD_URL/health` and `curl $PROD_URL/judiciary/health`.

**Use case:** Primary production target for public traffic.

**Requirements:**
- Kubernetes cluster with ingress controller.
- TLS termination at ingress.
- Container registry (GHCR: `ghcr.io/nkgoloL/eduboost-api`).
- Persistent volumes for logs / backups if needed.

---

## Render vs Kubernetes for production

| Concern | Render | Kubernetes |
|---|---|---|
| Operational complexity | Low | Medium |
| Cost (at scale) | Higher | Lower |
| Custom domain + TLS | Paid plan | Yes |
| Multi-region | No | Yes |
| Horizontal scaling | Limited | Full |
| Persistent storage | No | Yes |
| Recommended for | Staging / beta | Full production |

**Decision:** Use Render for staging and early beta. Migrate production
workload to Kubernetes before public launch or when Render free-plan
limitations become binding.

## Validation

```bash
# 1. Local compose starts cleanly
docker compose up --build -d
sleep 10
curl -f http://localhost:8000/health

# 2. Render Blueprint syntax is valid
# (requires Render CLI or manual validation in dashboard)

# 3. Kubernetes deployment manifest is valid
kubectl apply --dry-run=client -f docker/api-deployment.yml
```

---

## Consequences

### Positive

- Three-tier promotion model (local → Render → k8s) gives flexibility.
- Render provides quick staging without cluster management overhead.
- Kubernetes provides full control for production scaling and compliance.

### Negative

- Two production-like targets (Render + k8s) increase operational surface.
- Secret management is fragmented across GitHub, Render dashboard, and k8s secrets.
- Render free plan is not suitable for production traffic.
