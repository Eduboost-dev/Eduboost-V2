## Area 4 ‚Äî Error monitoring

### `app/frontend/src/lib/monitoring.ts` ‚Äî **new file**

PII scrubber + monitoring facade. Does **not** require an external DSN/API key to function ‚Äî it logs to console in non-production and is a no-op in production until a backend is registered.

**PII patterns scrubbed before any event leaves the app:**

| Pattern | Replacement |
|---|---|
| Email address | `[email]` |
| 13-digit South African ID number | `[za-id]` |
| SA phone number (`+27` / `0[6-8]x`) | `[phone]` |
| UUID v4 | `[id]` |

**Exports:**

| Export | Purpose |
|---|---|
| `scrubPii(value)` | Scrub a string |
| `scrubPiiFromObject(obj)` | Scrub all string values in an object |
| `registerMonitoringBackend(backend)` | Wire Sentry / OTel / custom at startup |
| `captureException(error, context?)` | PII-scrubbed error capture |
| `captureEvent(event)` | PII-scrubbed event capture |
| `captureTestEvent()` | Fire a known synthetic event to prove pipeline |

To connect Sentry later:
```typescript
import * as Sentry from "@sentry/nextjs";
import { registerMonitoringBackend } from "@/lib/monitoring";

registerMonitoringBackend({
  captureEvent: (e) => Sentry.captureMessage(e.message, e.level),
  captureException: (err, ctx) => Sentry.captureException(err, { extra: ctx }),
});
```

### `app/frontend/src/instrumentation.ts` ‚Äî **new file**

Next.js standard `register()` hook (runs once on server startup). On every boot:
1. Calls `getValidatedEnv()` ‚Äî fails closed if env is invalid.
2. In production: fires `captureEvent` init message.
3. On Node.js runtime: fires `captureTestEvent()` ‚Äî proves the monitoring pipeline is alive with synthetic (non-PII) data.
